# v1.0.35 build plan (Phase 0 deferred — Club tier slips to v1.0.36)

**Date:** 2026-05-25
**Status:** Approved scope — single-branch implementation, single mega-PR
**Supersedes (for v1.0.35 only):** parts of `docs/plans/2026-05-22-v1.0.35-monetisation-plan.md` that depend on Phase 0 (€1 `club_membership` IAP).

## Why this delta plan exists

The original 2026-05-22 v1.0.35 design ("The Club" — taster → €1 club → Pro) was scoped on the assumption Phase 0 (USER-only store/RC config) would land before code. Phase 0 is not happening this cycle. Plus, today's public reframe ([feedback_paywall_reframe_no_apology.md](../../../.claude/projects/-Users-StephenLear-Projects-locosnap/memory/feedback_paywall_reframe_no_apology.md)) locked the canonical "6 lifetime scans, no reset" position with three commenters publicly — the original plan's "taster = 3 scans" would now create a credibility hit.

This plan ships everything in the original v1.0.35 design that does NOT require the €1 club IAP, keeps the 6-lifetime-free tier intact, AND folds in the resilience + retention items that were going to slip to a separate interim build. The Club tier itself (€1 membership + Join screen + member-tier auto-opens + tier resolver + weekly counter) slots to v1.0.36 when Phase 0 lands. The original 2026-05-22 design doc remains canonical for that future release.

### Phase E — Offline-write queue for saveSpot

**Files:** new `frontend/services/spotQueue.ts`, modify `frontend/services/supabase.ts` `saveSpot`, listener wiring in app root

- New `services/spotQueue.ts`: pure queue logic with add / flush / peek over AsyncStorage key `locosnap_spot_queue`. Queue payload format: `{ spotPayload, queuedAt }`.
- `saveSpot`: on network error, queue the payload and return success to the caller (UI should show "Saved" — the queue is invisible to the user).
- Connectivity listener (`@react-native-community/netinfo`): when network returns online and queue is non-empty, flush in FIFO order. Bounded retry per item (e.g. 3 attempts), then mark terminal failure.
- Sentry: `captureError` only on terminal failures (validation rejection, persistent 5xx). Do NOT capture transient network drops — that's what the queue is for.
- Pure-logic tests for queue add/flush/peek (matches project's ts-jest convention).
- Closes the `saveSpot Network request failed` Sentry pattern at root (5 events / 3 users / 13 days currently; below escalation threshold but the queue is the structural fix).

**Commit message:** `feat(supabase): offline-write queue for saveSpot — queue on network failure, flush on reconnect`
