# v1.0.35 Monetisation ("The Club") Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship the three-tier "Club" monetisation model — free taster → €1 one-time club membership → Pro subscription — with a tier-aware home card, peak-moment auto-open paywalls, and a restructured Pro paywall.

**Architecture:** Tiers are derived from RevenueCat entitlements (`member`, `pro`) plus scan counts. The €1 membership is a non-consumable IAP behind a new `member` entitlement; Pro stays the `pro` entitlement. A member's weekly scan allowance is tracked server-side in Supabase. New "Join the Club" screen; existing `paywall.tsx` restructured for Pro.

**Tech Stack:** React Native + Expo (TypeScript), Expo Router, Zustand, RevenueCat (IAP/subscriptions), Supabase (Postgres + Auth).

**Spec:** `docs/plans/2026-05-22-monetisation-design.md` — read it first; it is authoritative. This plan turns it into tasks.

**Note for the implementer:** this plan gives exact files, deliverables, test expectations and commit points. For each code task, **read the named file(s) first** — paywall.tsx, the stores, and the scan flow are large and must be understood before editing. Do not fabricate; match existing patterns.

## Phase 1 — Tier model & scan gating

### Task 1.1 — Tier resolver
**Files:** Create `frontend/services/tier.ts`; Test: `frontend/__tests__/services.tier.test.ts`.
Pure function `resolveTier({ hasPro, hasMember }) => "taster" | "member" | "pro"` — `pro` wins, then `member`, else `taster`. Pure-logic test (matches the project's ts-jest convention). TDD: failing test → implement → pass → commit.

### Task 1.2 — Member weekly-allowance counter (Supabase)
**Files:** Create `supabase/migrations/016_member_weekly_scans.sql`; Modify the `Profile` type + `services/supabase.ts`.
Add columns to `public.profiles`: `member_scans_week int not null default 0`, `member_week_start date`. (Column ADD on an existing table — the post-2026-10-30 grant template does **not** apply; existing grants are retained.) Logic: on a member scan, if `today >= member_week_start + 7` reset counter to 0 and `member_week_start = today`; else increment. Read the existing `daily_scans_used` handling first and mirror the pattern.

### Task 1.3 — Apply tier to scan gating
**Files:** Modify `frontend/app/(tabs)/index.tsx` (the `canScan` / `scansRemaining` / `MAX_FREE_SCANS` logic) and `frontend/store/authStore.ts`.
Taster → 3 lifetime scans; member → weekly allowance (~4, a single named constant `MEMBER_WEEKLY_SCANS`); pro → unlimited. Read the current gating block before editing. Test the limit logic where it can be extracted as pure logic.

**Commit** after each task.
