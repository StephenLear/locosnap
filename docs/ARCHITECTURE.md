## Overview

LocoSnap is a mobile app that identifies trains from photos using AI. Users take a photo, the backend identifies the train and returns class, operator, specs, rarity, historical facts, and an AI-generated blueprint illustration.

### Hardening (2026-05-13 audit)

- **Production env invariants** — `assertProductionConfig()` in `src/config/env.ts` crashes the dyno on boot if any of `ANTHROPIC_API_KEY|OPENAI_API_KEY`, `SUPABASE_URL`+`SUPABASE_SERVICE_ROLE_KEY`, `REVENUECAT_WEBHOOK_SECRET`, or `ADMIN_SECRET` is missing when `NODE_ENV=production`. Dev/test remain permissive.
- **RevenueCat webhook auth** — in production, the route hard-fails with 503 if `REVENUECAT_WEBHOOK_SECRET` is unset (closing the silent-grant-Pro hole). Token comparison uses `crypto.timingSafeEqual`.
- **Identify rate limits** — two layers: anonymous IP limiter (20/hr, skipped for authenticated requests to avoid CGNAT collisions; pre-existing) **plus** an authenticated per-token limiter (60/hr, added 2026-05-13) so a single compromised JWT cannot drain Vision/Replicate spend. - **CORS** — function origin matcher accepting exact-allowlist origins plus `exp://` / `exps://` schemes (the previous exact-string `"exp://"` entry was dead).
- **Frontend auth storage** — Supabase session moved from `AsyncStorage` to `expo-secure-store` (iOS Keychain / Android Keystore) via a chunked adapter at `frontend/config/secureStorage.ts` (1800-byte chunks to work around iOS's 2 KB per-value limit). **AI call configuration:** Temperature is set to `0` on **all four** AI calls (vision, specs, facts, rarity) across both Claude and OpenAI paths — re-confirmed 2026-04-26 commit `27cf25a` after a regression introduced by the 2026-04-25 Sonnet → Haiku flip exposed that `trainFacts.ts` and `rarity.ts` had been running at default temperature (the audit was missed when the model identifiers were swapped). **Prompt caching strategy (cost) — 1-hour TTL since 2026-06-01:** All four AI calls cache their system prompt via `cache_control` on the `system` block. As of 2026-06-01 the TTL is **`{ type: "ephemeral", ttl: "1h" }`** (1-hour, GA, no beta header) on vision (`TRAIN_ID_PROMPT`), specs, facts, and rarity — changed from the default 5-minute TTL. Reason: the vision system prompt has grown to **~85,000 input tokens** (~272,800 chars / ~68K tokens of accumulated per-class disambiguation rules) and is re-read on every scan. A 2026-06-01 cost audit found per-scan cost flat (~$9.45/day, ~$283/mo, ~$0.18/scan) since the 2026-05-23 hotfix despite a 98.1% cache read ratio, because on quiet/sparse-traffic days the 5-minute cache expired between scans and re-paid the full cache **write** (~$0.32) instead of a read (~$0.026) — confirmed by Sonnet write-amortization of only 2.74×. The 1-hour TTL keeps the prompt warm across the typical intra-cluster scan gaps. Cost ratios: cache read = 0.1× base input; 5-min write = 1.25×; 1-hour write = 2× — the higher write cost is more than offset by paying far fewer writes. SDK note: `@anthropic-ai/sdk@^0.39.0` `CacheControlEphemeral` type omits `ttl`, so the literal is cast `as any` at each call site (SDK forwards the field to the API unchanged). **Prompt trim / two-stage split — TESTED 2026-06-01 and REJECTED (do not re-attempt without new evidence).** Built a lean core + lazy long-tail (23 rare rules moved out, ~20% off the prompt) and eval-tested it: it regressed accuracy (lean-core 27/36 vs 28/36; SW1001 → "Class 08"). A focused-prompt PoC proved prompt SIZE is not the cause of the misIDs, and stage-2 cannot recover a removed load-bearing rule (stage-1 returns the wrong class, which never triggers the right family lookup). Reverted. The two high-certainty cost wins remain the 1h TTL + canonicalisation above. **Regression gate:** `npm run vision:eval` (`backend/src/scripts/visionEval.ts`) runs 36 labelled photos from `~/Desktop/train photos/` through the live vision path and scores vs expected class with mustNot guards — run it before ANY future prompt-structure change. Needs `ANTHROPIC_API_KEY` in `backend/.env` (dev-only, ~$1/run). Current baseline 28/36; the 8 known misIDs (ICE4→408, BR159→193, Mireo→563, Thumper→411, …) are model/image limits, NOT prompt-fixable.

**Class-name canonicalisation (`classNames.ts`, 2026-06-01):** Vision emits the same physical class under several spellings (`BR 232` vs `DB BR 232`, `DB Class 628` vs `BR 628`). `canonicaliseClass()` collapses these to one canonical form at `parseTrainResponse` (vision.ts) — the single choke point both the Claude and OpenAI paths share — so the cache key (`language::class::operator`), specs, facts, rarity, and the displayed class are all consistent. Conservative: only `DB`-prefixed German variants and `Baureihe → BR` are rewritten (plus an explicit alias map, currently just `DR BR 132 → BR 232`); UK `Class N`, `ICE 3`, `ÖBB`, Polish classes are untouched. This fixes two things found in the 2026-06-01 scan-distribution audit: cache fragmentation (same train cached under multiple labels → wasted AI spend) and **non-deterministic rarity** (a new label = a fresh cache-missed rarity call → the "same train, different rarity tier" bug reported by tester Leon). Idempotent; no cache version bump (old non-canonical entries simply orphan and re-compute once). Operator-string variance is a separate, not-yet-addressed fragmentation source.

## 6. Blueprint Task Store + Train Data Cache — Redis

| Property | Value |
|----------|-------|
| Production | Upstash Redis |
| Local Dev | In-memory Map (automatic fallback) |
| Blueprint tasks | Stores async blueprint generation task status (key: `task:<id>`) |
| Train data cache | Stores specs/facts/rarity/blueprints per train class+operator (key: `traindata:<class>::<operator>`) |

**Cache architecture (2026-03-22):** The train data cache was previously written to the local filesystem (`train-cache.json`) which is wiped on every Render deploy. Migrated to a two-level cache:
- **L1** — in-memory `Map` (fast, resets on server restart)
- **L2** — Upstash Redis with 30-day TTL (persistent across deploys)

Cache entries are lazy-loaded from Redis on first access. `trainCache.ts` functions (`getCachedTrainData`, `setCachedTrainData`, `setCachedBlueprint`) are all async. Saves ~84% of AI costs on repeat scans (£0.005 cached vs £0.031 fresh).

**Cache key format (post-2026-05-23 hotfix `76660c3`): `{language}::{class}::{operator}` — no version prefix.** The previous `v9::` / `v12::` global version prefix was removed when the cache-version bump pattern was retired (see below). Existing v8-v12 keys in Redis become orphans and TTL out naturally over 30 days.

**Per-class invalidation (replaces global CACHE_VERSION bumps).** Before the 2026-05-23 hotfix, fixing one wrong-identification class meant bumping a global `CACHE_VERSION` constant, which wiped the entire 30-day Redis cache for every class on every fix. v8 → v9 → v11 → v12 in 7 days (May 12-19) = four full cache wipes, each forcing every subsequent scan back through the full 4-call AI pipeline for 24-48h until the cache rebuilt. This was the single biggest unforced cost leak in the May 2026 audit (~$0.20/scan during the bumpy period vs ~$0.06/scan predicted post-fix).

New rule: when a fix ships that invalidates one class's specs/facts/rarity, add ONE entry to `CLASS_INVALIDATIONS` in `trainCache.ts` keyed by the normalised class name and the cut-off timestamp. Cache entries for THAT class cached before the timestamp are treated as misses on next read; every other class's cache survives. Variant-coverage rule (added 2026-05-24 after the ÖBB 4020 stage-3 retrofit): every `KNOWN_SPECS` lookup key for a class needs a matching `CLASS_INVALIDATIONS` entry, otherwise pre-fix Redis cache leaks through under any uncovered variant. See [`memory/backend_cache_invalidation_pattern.md`](file:///Users/StephenLear/.claude/projects/-Users-StephenLear-Projects-locosnap/memory/backend_cache_invalidation_pattern.md) for the 4-step checklist.

**Cost economics (2026-05-24 evening measurement, day-1 post-hotfix):** marginal ~$0.06/scan + ~$4-5/day fixed daily overhead. At 365 scans/day → blended $0.072/scan. At 1000 scans/day → blended $0.065/scan. Sonnet 4.6 cache read ratio 96.0%, Haiku 4.5 99.3%. Investigation queued: identify the $4-5/day fixed overhead (failed-vision retries, anonymous pre-spot calls, or warmup) via Anthropic Logs on a low-volume day. **Supabase Auth → Resend (discovered 2026-05-18):** Supabase Auth's emails — sign-in OTP codes ("Your LocoSnap Sign-In Code"), magic links, confirmation emails — are routed through this same Resend account via the Supabase SMTP / custom-domain configuration. From address: `"LocoSnap" <noreply@locosnap.app>`. **These emails count against the Resend send quota.** Visible in the dashboard export as rows with a different `api_key_id` from the backend/script sends (Supabase has its own API key entry in the Resend dashboard). Practical implication: when sizing a backfill or estimating remaining quota, add expected Supabase Auth volume on top of intentional sends. With the unlimited-tier upgrade (above) this no longer blocks anything, but the export's daily total will always include sign-in traffic — don't be surprised by it.

**Resend suppression list — silent drop gotcha (discovered 2026-05-18):** Resend maintains an account-wide suppression list of addresses that previously hard-bounced, complained, or unsubscribed. **When you attempt to send to a suppressed address, the API returns `200 + message_id` (looks like success), but Resend never delivers the email.** The only way to detect this is the dashboard's per-email status: suppressed sends show `last_event: suppressed` in the export CSV. The 2026-05-18 batch 2 hit this on two addresses (`tobias.trostmann@web.de`, `hobbybundesbahner@outlook.de`) — both had a same-person variant at a neighbouring row that delivered fine, so no one was fully missed, but had they not signed up twice they would have been silently dropped. **Operational rule going forward:** for any non-trivial batch (>20 sends), pull the Resend export afterwards and check `last_event` rather than trusting the API-acceptance count from the send script. Cross-reference the export with the recipient list to find suppressions. Suppressions cannot be retried — Resend will reject any further send attempts to that address until manually un-suppressed in the dashboard.

**Resend MCP server — developer tooling (added 2026-05-30):** The official `resend-mcp` server (`npx -y resend-mcp`) is registered at **project scope** in `.mcp.json` (repo root, **gitignored** — it holds the API key in plaintext, a second copy outside `backend/.env`; never commit or un-ignore it). Tools surface as `mcp__resend__*` after a Claude Code session restart (not mid-session). This is the practical workaround for the silent-drop gotcha above: the MCP exposes **list emails** and **get email**, both returning the per-message `last_event` field — so post-batch suppression checks can be done programmatically instead of pulling the dashboard CSV. Limitation: there is **no suppression-list tool**; outcomes are read per-email via `last_event`, and un-suppressing still requires the dashboard. Installed MCP-only (not the full Resend Claude Code plugin) to avoid the unused React Email / agent-inbox skills.

**Resend API gotcha — Cloudflare User-Agent block (discovered 2026-04-11):** The Resend API now sits behind Cloudflare and rejects requests with the default `Python-urllib/3.x` User-Agent with `403 Forbidden` and Cloudflare error code `1010` ("the owner of this website has banned your access based on your browser's signature"). Any script calling the Resend API MUST include a custom `User-Agent` header. Working header set used 2026-04-11 to send 5 tester check-in emails:

```
Authorization: Bearer <RESEND_API_KEY>
Content-Type: application/json
User-Agent: LocoSnap-TesterMailer/1.0 (curl-equivalent)
Accept: application/json
```

Without the User-Agent override, all 5 emails returned `403 / error 1010` and 0 sent. With the override, all 5 sent successfully on the next attempt. Apply the same pattern to any future Python/urllib-based Resend calls. `curl` and Node SDKs are not affected because they send their own non-empty User-Agent by default.

## 17. User Flow (Scan-First, No Guest Mode)

```
1. App opens → user goes directly to scanner (no auth gate)
2. Trial banner shows: "3 free scans to try — sign up to save your collection"
3. User scans (up to 3 times, counted in AsyncStorage)
4. On scan 4: "Create Your Free Account" prompt — sign-up gate
5. User creates free account → 3 lifetime scans, cloud sync, leaderboard
6. User upgrades to Pro → unlimited scans, all blueprint styles
```

- `PRE_SIGNUP_FREE_SCANS = 3` — trial scans before sign-up required
- `MAX_FREE_SCANS = 3` — lifetime limit for free accounts (changed from `MAX_MONTHLY_SCANS = 10` in v1.0.19)
- `PRE_SIGNUP_SCANS_KEY = "locosnap_presignup_scans"` — AsyncStorage key

## 18. Data Flow (Single Scan)

```
1. User takes photo
2. canScan() checked — trial/monthly limit enforced
3. App uploads photo to POST /api/identify
4. Backend pre-warms on app mount (healthCheck) to avoid cold start
5. Vision API identifies the train (Claude or GPT-4o)
6. Cache check — if train class+operator seen before, skip steps 7-9
7. Specs + Facts + Rarity fetched in parallel (Claude or GPT-4o)
8. Results cached in Redis (30-day TTL)
9. Blueprint generation starts async → returns taskId
10. Train data returned immediately to app
11. App displays results + polls GET /api/blueprint/:taskId
12. Blueprint completes → uploaded to Supabase Storage
13. Spot saved to Supabase (spots table) — authenticated users only
14. XP awarded, streak updated, achievements checked
15. Leaderboard updated
```

- "Free on App Store" in yellow (#FFFF00) Impact text
- "Coming soon to Android" in yellow (#FFFF00) Impact text
- Dark background (#0d0d0d)

**Text overlays (ASS subtitle format for ffmpeg):**
- Font: Arial Black (Impact-weight, bold -1)
- Colour: yellow (`&H0000FFFF` in ASS AABBGGRR format = #FFFF00)
- Outline: black (`&H00000000`), 6px — essential for legibility over bright footage

- Alignment: 2 (bottom-centre) unless overriding with `\an` tag
- Two lines maximum — keep it punchy, not explanatory
- No time claims for blueprint generation (takes up to 60 seconds in the app)
- ASS style reference (720p portrait): `Style: Impact,Arial Black,110,&H0000FFFF,&H000000FF,&H00000000,&HA0000000,-1,0,0,0,100,100,0,0,1,6,2,2,30,30,100,1`
- **Character limit per line (tested 2026-04-15 on BR 143 build)**: At 110px Arial Black width 720px with 6px black outline, the safe maximum is **10 characters comfortably, 11 tight but OK, 12 or more overflows the frame edges**. Tested fits: "DIE BR 143" (10) ✓, "646 GEBAUT" (10) ✓, "FAST WEG" (8) ✓, "STIRBT AUS" (10) ✓, "DAS DR-ERBE" (11) tight but OK. Tested overflows: "VERSCHWINDET" (12) ✗, "IS VANISHING" (12) ✗ — both had to be replaced during the BR 143 build cycle. Always count characters before finalising overlay text; plan shorter alternatives for long German compound words. Earlier handover notes claimed a "9-char safe limit" which is outdated — actual tested limit is 10–11.

- Never open on the scan UI or app chrome
- Blueprint reveals go after the footage cut, not before
