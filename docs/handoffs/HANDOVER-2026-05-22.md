173/173 backend tests pass, typecheck clean. Pushed; Render auto-deployed.

### 2. Wrong-ID report table investigation → frontend photo-capture fix (`d43ab90`)

Traced where the in-app "Wrong ID" report goes: Supabase table `public.wrong_id_reports` (migration 012). Found **`photo_url` is null on every row** — `submitWrongIdReport` accepts a `photoUrl` param but no caller ever passed it; the misID triage queue was largely unactionable. A live query of the table showed ~60 reports in 6 days, zero with photos, and JonnySeagull's two steam misIDs sitting there with corrections but no images.

Fix: `submitWrongIdReport` now accepts a `photoUri` — a remote URL is stored directly, a local scan URI is uploaded to Supabase Storage via `uploadPhoto` when a userId is present. The card-reveal "Wrong ID" tap and the low-confidence-decline path now pass the scanned photo. Anonymous users still file a report without a photo (no regression). New test added; 154/154 frontend tests pass.

### 3. Frontend — gallery scans use EXIF GPS for location (`d43ab90`)

RailUK tester sabanda reported a photo taken at Reading being labelled Bournemouth — gallery scans logged the device's current location, not the photo's. New `parseExifGps` helper in `app/(tabs)/index.tsx` reads GPS from the picked photo's EXIF (iOS nested `{GPS}` dict / Android flat `GPS*` keys, hemisphere-ref sign handling). For `captureSource === "gallery"`, the photo's EXIF GPS overrides device location. `photoAccuracyM` deliberately left as the device reading so verification tiers are unchanged.

### 4. Frontend — in-app Pro rescue prompt (`9277525`)

New `components/ProRescuePrompt.tsx` — a one-time dismissable card on the scan screen for Pro users who never logged a spot (`is_pro && last_spot_date == null`), the cohort from the 2026-05-17 RevenueCat audit. EN/DE/PL strings under `scan.proRescue`, dismissal persists in AsyncStorage. Chosen over a server push after confirming there is no server push infrastructure (`profiles.push_token` column does not exist, no send path, Android push disabled) — an in-app prompt reaches both platforms.

## Key Decisions

- **Class 66 rule: fleet number primary, livery secondary.** Livery-based operator ID is fundamentally unreliable for the 66 (transfers, debranding, Heavy Haul Rail in Freightliner colours). Fleet number is the only dependable cue.
- **Pro rescue = in-app prompt, not server push.** No push token storage, no send path, Android push disabled — a push would be iOS-only and miss the cohort. In-app reaches both platforms.
- **ICE / BR 403 cluster left parked.** 4 report-table hits but it needs the actual photos; the new wrong-ID photo capture will start producing them once v1.0.34 ships.
- **Corrective ad comments are a positive, not a problem** (user steer) — they drive TikTok algorithm distribution. Saved as memory `feedback_ad_comment_corrections_positive.md`.
- **v1.0.34 bundles all three frontend changes** into one build rather than separate builds.

## Lessons Learned & Gotchas

- **`wrong_id_reports` has no triage surface** — no SELECT policy, no endpoint, no notification. Reports are only visible via a manual Supabase SQL query. ~60 sit unread; only 3 have a `spot_id` (photo recoverable from Storage). Once v1.0.34 ships, photos will attach going forward.
- **No server push infrastructure exists.** `profiles.push_token` column is absent (saves silently fail), no send path, Android push hard-disabled (Android-16 crash). A real rescue push is a separate, larger project.
- **EAS build credits at 95%** for this billing cycle — further builds are pay-as-you-go until reset.
- **eas-cli 18.5.0 is too old** for the current EAS API — use `npx eas-cli@latest`. Consider a global update.
- **`ad_assets/` is untracked by convention** (per memory `ad_assets_endcards.md`) — expected, not a blocking issue at handover.
- The native frontend changes (EXIF GPS parse, Storage upload) are **verified by tests + typecheck only, not on-device** — must be checked on the v1.0.34 build.
