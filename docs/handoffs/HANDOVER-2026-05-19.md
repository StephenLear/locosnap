### The actual fix (commit `d1cd710`)

**Root cause:** Expo's `expo.locales` block in `frontend/app.json` processes JSON locale files for **both** platforms. On iOS it generates `<locale>.lproj/InfoPlist.strings`. On Android it generates `res/values-b+<locale>/strings.xml`. The iOS NSx Info.plist permission keys (NSCameraUsageDescription, NSPhotoLibraryUsageDescription, NSPhotoLibraryAddUsageDescription, NSLocationWhenInUseUsageDescription) are inherently iOS-only ‚Äî they have no meaning on Android. But Expo's locale plugin doesn't filter by platform, so they end up in Android's localized strings.xml with no matching entry in the unqualified `values/strings.xml` (Android Lint's "default locale" definition). Lint's `ExtraTranslation` rule then fails the release build with 4 errors per locale per key.

**Why "add an `en` entry" didn't work:** Android Lint defines "default locale" strictly as the unqualified `values/strings.xml` directory. Anything with a BCP-47 suffix (including `values-b+en/`) is a translation, not a default. There is no way to produce an unqualified `values/strings.xml` via `expo.locales` alone ‚Äî every entry generates another locale-suffixed file.

**Real fix:**
- New config plugin `frontend/plugins/withDisableExtraTranslationLint.js` (mirrors the existing `withSentryDisableUpload` pattern), uses `withAppBuildGradle` from `@expo/config-plugins` to inject `lint { disable 'ExtraTranslation' }` into the generated `android/app/build.gradle` `android { ... }` block during prebuild. Only `ExtraTranslation` is suppressed; all other lint checks continue to run in vital-release mode.
- `frontend/app.json` updated: registered the new plugin in `plugins[]`, removed the orphaned `en` entry from `locales` (EN strings are already the default in `ios.infoPlist`).
- The NSx keys still get generated as Android `values-b+<locale>/strings.xml` resources ‚Äî they are dead resources on Android (zero code paths reference them) but the lint rule no longer fails the release build.

**Verification before push:**
- `npx expo prebuild --platform android --clean` from main checkout
- `lint { disable 'ExtraTranslation' }` injected at line 88 of generated `android/app/build.gradle`, immediately inside `android { }`
- `values-b+de/strings.xml` and `values-b+pl/strings.xml` retain DE/PL NSx strings (iOS localisation unchanged)
- Could not run `:app:lintVitalRelease` locally (no Android SDK on this machine) ‚Äî relied on documented rule-suppression mechanism which is stable in AGP

### Two preventable session-level mistakes
1. **Shipped a fix to a release-blocking lint failure without local reproduction.** The previous session's commit `26361e8` added an `en` entry to `expo.locales` without ever testing the resulting prebuild. Build 22 confirmed the fix was wrong only AFTER 13 minutes of EAS time + a paid credit. Local repro (5 min, no credit) would have caught it immediately.
2. **Triggered `eas build` from an unpulled checkout.** Build 23 was meant to be the d1cd710 fix but EAS uploaded the stale local working tree, not origin/main. Should have run `git log -1` before triggering. Now codified in `project_status.md` rules section.

### Process recommendations going forward
- For any release-blocking gradle / lint failure: reproduce locally via `npx expo prebuild --platform android --clean` and inspect generated files BEFORE triggering an EAS build. If Android SDK installed, also run `cd android && ./gradlew :app:lintVitalRelease`.
- Before triggering `eas build` from `frontend/`: confirm `git log -1` HEAD includes the intended fix. EAS uses the local working tree, not the remote.
- Use `--clear-cache` flag on EAS build whenever a previous build of the same fingerprint failed ‚Äî bypasses cached build artifacts.
- After triggering, immediately `eas build:view <id>` and verify `gitCommitHash` matches expected AND `fingerprint.hash` differs from any prior failed-build fingerprint. Only walk away after this check.
