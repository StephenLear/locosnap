## Lessons Learned & Gotchas

- **`.claude/worktrees/` MUST be in `.gitignore` before creating any worktree.** Otherwise the worktree's working tree leaks into the parent repo's `git status` and risks accidental commits of generated files. Now permanent. Future: every new tooling directory triggers a `.gitignore` audit.
- **Branch off the latest in-flight release, not `main`.** v1.0.24 TSC fixes lived on the v1.0.24 branch and weren't on main. v1.0.25 worktree had to be re-pointed at the v1.0.24 head. Generalisation: until release branches are merged into main, downstream feature work branches off the release branch.
- **5-day-stale memory files about "queued" work.** Memory decays on operational state (git status, queued work, build state). Always cross-check against the changelog or `git log` before acting on a memory claim that says "pending" or "queued".
- **Tester emails — replies focus on the fix.** Don't apologise for our internal timing. State the diagnosis + ship date. Honest and crisp beats apologetic and apologetic-AND-honest.
- **Design doc nomenclature can drift from the codebase.** Brainstorm sessions are creative; they should be reconciled against existing types/constants before migration drafting. The migration was the forcing function that surfaced the VERIFIED/PERSONAL/UNVERIFIED → `verified-live`/etc. mismatch.
- **`eas build --platform all --no-wait` + foreground monitoring.** Yesterday's lesson held: the `--no-wait` flag uploads + queues + exits cleanly. Monitoring via `eas build:list --json` then submitting via `eas submit --latest` worked end-to-end without ever getting stuck.
- **EAS Submit with `releaseStatus: draft`** — the AAB lands as a draft on Play Console's Production track. User must navigate to the existing draft to edit release notes, NOT click "Create new release" (which makes a separate empty release). Documented in 2026-05-02 handover; held this session.
- **Apple + Google approval timing**: today both stores approved same day (afternoon for Google, evening for Apple). Yesterday Google was hours, Apple overnight. **Don't assume parallel timing**.
- **Subagent-driven for trivial mechanical tasks is overkill.** The two-stage review (spec compliance + code quality) on a one-line constant change is theatre. Use the discipline for real-logic tasks (Section B onward), use direct execution for mechanical paste-SQL / bump-const tasks.
- **Single-question brainstorming generates locked decisions cleanly.** 14 locked decisions across 2 hours. The streak-freeze instinct came from the user mid-flow and slotted in cleanly because the structure didn't lock down before all the questions were asked.
