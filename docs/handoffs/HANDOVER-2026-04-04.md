## Lessons Learned & Gotchas

- **Native exceptions bypass JS catch blocks**: `getExpoPushTokenAsync` on some Samsung/Android 12+ devices throws before the JS bridge can intercept. The fix is wrapping at the function body level, not just adding `.catch()` at the call site in `_layout.tsx`. Always wrap Expo native module calls in their own try/catch when building the function, not just at invocation.
- **The multi-provider architecture was startup-time only**: `if (config.hasAnthropic)` selects the provider once at boot. A runtime billing error fallback required a separate catch block for HTTP 402. Do not confuse configuration-time provider selection with runtime error handling.
- **EAS Build clones from git — untracked files do not exist**: `eas.json` is committed (confirmed). Any future patch files or config referenced by build scripts must be committed or they silently won't apply in the build.
- **ET22 max speed is 120 km/h not 140**: Polish TikTok commenter corrected this. The vision prompt disambiguation rule currently says 140. Fix queued for v1.0.12.
- **EU07A has a different front end from EU07**: A TikTok commenter identified EU07A-001 (ex EU07-495) — the app returned EU07 not EU07A. The modernised EU07A has a visually distinct cab. Worth a prompt refinement if the pattern repeats in comments.
- **IP rate limit uses the exit node IP for VPN users**: The Finnish tester uses Windscribe VPN (confirmed from wrong debug log). VPN users share exit node IPs — if a popular exit node hits 20 scans/hour, legitimate users on that VPN could be rate-limited. Low risk at current scale, worth noting for the future.
- **Polish audience is 24.9% of TikTok viewers this week**: ET22, EU07, PKP content consistently outperforms UK content for this account. The algorithm is routing Central European rail fans. Content strategy should follow the data.
