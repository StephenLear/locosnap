## Lessons Learned & Gotchas

- **Any WIKIDATA_CORRECTIONS addition for a class that may have been scanned before MUST bump `CACHE_VERSION` in the same commit.** The cache stores at `(class, operator)` granularity keyed by version. Existing cached entries with the old wrong specs continue to serve until the version bumps. Lost ~5 min today re-investigating why Rc6 still showed 200 km/h post-deploy. Adding this rule to the backend correction workflow.
- **Anthropic SDK uses `err.status`, not `err.statusCode`.** If you write error-handling code that branches on HTTP status, check both fields (or just `err.status ?? err.statusCode`). Native Node `fetch` errors and Axios errors also differ. The new `isUpstreamOverload` helper covers all three error-shape conventions.
- **Cache key format `v<n>::<lang>::<class>::<operator>`** all lowercase, all trimmed. Cache is per-language: EN and DE entries are separate. Bumping the version orphans all stale Redis entries automatically — they expire naturally on TTL or get reclaimed by LRU. Not deleted, just unreachable.
- **Render deploys auto-fire on push to backend-matched paths even if CI fails.** This is a feature (production stays current even if a test is stale) but also a footgun if a test failure represents a real bug. Worth double-checking the CI output when a backend push lands, even if the deploy succeeded.
- **expo-location's `coords.accuracy` is a float, not an integer.** When ingesting any sensor data into a typed database column, verify the column type matches the JS type. Same gotcha class applies to: GPS coordinates, sensor readings, motion data, audio sample rates, anything that returns a Number from a native API. Audit pattern in `feedback_supabase_silent_persistence_failures.md` extended accordingly.
- **Apple approves on Tuesday too**, not just overnight. v1.0.30 build 52 was submitted ~24h before approval — typical 24-48h Apple Review turnaround. Once the binary's in, the gate timer is reasonably predictable.
- **Samuel partnership pattern is "low-friction, fast-response"**: every touchpoint in the past 24h has had same-day positive return. Pattern worth applying to future creator outreach — DM with specific narrow ask + clear credit terms + heads-up beat on each ad post.
- **Polish ad-copy critique pattern**: technical correctness matters to Polish rail fans more than I'd predicted. "Następca" (successor) was the wrong noun for E6ACT vs ET43 — they're "starszy kuzyn" (older cousin) / same Newag Dragon family. Future Polish ad copy should default to family-relationship language ("Dragon 1 vs Dragon 2", "z tej samej rodziny") rather than generational/successor framing.
