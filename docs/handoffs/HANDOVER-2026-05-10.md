### P1 platform-wide silent data-loss bug — DISCOVERED + FIXED late-evening

**Discovery:** Steph (UK heritage Pro tester, evangelist tier) reported scans missing from history. Diagnostic trail with her cooperation across multiple screenshot rounds:
- 23:11 IMG_4700: Class 37 "Barbara Arbon" 9m-ago entry visible in history, unique=126 / total=200
- 23:29 IMG_4702/3: Class 37 dropped from history list (pull-to-refresh didn't recover it), unique=126 / total=200
- 23:30 IMG_4704 (post force-stop + reopen): Class 37 still gone AND **Unique Classes dropped 126 → 125** while Total Spots stayed at 200

The unique-count drop after force-stop was the smoking gun — proved local-only entries (failed-to-persist scans) were being wiped on cloud rehydration. Same pattern explained why Steph's entire Saturday 2026-05-09 Chastleton session (6+ misIDs) was absent from history too.

**Diagnosis via code + Supabase SQL:**
1. Traced `saveSpot` in `frontend/services/supabase.ts` — silently swallows errors with `console.warn`, returns null
2. Traced `loadHistory` in `frontend/store/trainStore.ts:242-246` — cloud-replace strategy destroys local-only entries with no merge
3. Queried Supabase: Steph has 210 spots / 85 unique classes server-side; ZERO spots since 2026-05-09 00:00 UTC
4. Manual INSERT via SQL editor returned the exact error: `ERROR: 23502: null value in column "verification_tier" of relation "spots" violates not-null constraint`
5. Schema audit confirmed: `verification_tier` is `text NOT NULL DEFAULT null` — every other NOT NULL column on `spots` has a default
6. Distribution audit confirmed: 1565 existing rows = `personal`, 23 = `verified-recent-gallery` — so `personal` was the historical default before something stripped it

**Root cause confirmed:** Migration 015 (applied 2026-05-08) or some related v1.0.29-era schema change left `verification_tier` as `NOT NULL` without a default. The v1.0.29 frontend's `saveSpot` payload doesn't include `verification_tier`. Every insert from every v1.0.29 client has been silently rejected since 2026-05-09 (or earlier). The frontend's silent-error-swallow + cloud-replace `loadHistory` combination made the failures invisible until Steph's precise capture exposed them.

**Fix shipped same night via Supabase dashboard SQL editor** (no migration file, no app rebuild needed):
```sql
ALTER TABLE spots ALTER COLUMN verification_tier SET DEFAULT 'personal';
```

**Verification end-to-end** (driven via Chrome MCP + Supabase SQL editor):
1. ALTER returned `Success. No rows returned` ✓
2. Test INSERT for Steph's `user_id` WITHOUT specifying `verification_tier` returned a row with `verification_tier=personal` ✓
3. DELETE of test row succeeded ✓
4. Follow-up SELECT confirmed 0 rows for test id (database clean) ✓

**Impact:** every v1.0.29 client `saveSpot` from this moment forward persists correctly. Steph's specific lost scans (2026-05-09 Chastleton + 2026-05-10 23:11 Class 37) cannot be recovered server-side — they never persisted. If she has the photos in her camera roll, re-scanning brings them back.

**Architectural defects that remain** (v1.0.30 backlog):
1. `saveSpot` / `upsertTrain` / `deleteSpot` / `updateSpotBlueprint` all silently swallow errors with `console.warn`. Need `Sentry.captureException` with failing payload as context.
2. `loadHistory` cloud-replace destroys local-only entries. Need merge logic that preserves local-only items not yet present in cloud.
3. No retry queue for failed inserts. Need offline-first retry with exponential backoff.
4. `saveSpot` should pass `verification_tier` explicitly in the payload as defense-in-depth (column default is fallback, not primary).

## Lessons Learned & Gotchas

- **RC Play prepaid base plans are the v6+ SDK feature, NOT autorenew.** Counterintuitive but verifiable: in RC's offering edit view, prepaid `:monthly` carries the v6+ warning, autorenew `:monthly-autorenew` is marked "Backwards Compatible: Yes." The mental model "autorenew is the modern thing" is wrong — autorenew is the universal default; prepaid is the new Play Billing feature. This was the root of the failed swap attempts (we kept thinking autorenew should be the v6+ primary; RC's UI structure forces the inverse).
- **RC's "Current base plan" label is a per-product self-identifier, not a global setting.** Every product detail page (`pro_monthly:monthly`, `pro_monthly:monthly-autorenew`, etc.) shows the SAME Base Plans table and labels its own row "Current". Don't interpret the label as evidence that one base plan is the served default — that lives in the offering's package config.
- **RC's "Default" = SDK's "Current".** When you hit the `...` menu next to an offering and want to set it as the SDK's `getOfferings().current` target, the action is labeled "Make Default" — not "Set as Current" or "Make Active". Same thing, just RC-specific terminology.
- **`offerings.current` in [paywall.tsx:183](frontend/app/paywall.tsx) auto-picks whatever's marked Default in RC.** No app rebuild needed when changing the Default offering — reads next time the paywall opens. The `blueprint_credits` lookup at paywall.tsx:191 is by-id and unaffected.
- **yt-dlp embeds the YouTube channel name and original URL in mp4 metadata as `TAG:artist` and `TAG:comment` by default.** ffprobe extracts both in one shot. This is the unambiguous provenance check on any past ad: `ffprobe -v error -show_entries format_tags -of default=noprint_wrappers=1 file.mp4 | grep -E "TAG:(artist|comment|title)"`.
- **The BOB ambiguity in DE rail.** Two different BOB operators, both Transdev-branded but in different regions running different rolling stock: BOB Bodensee-Oberschwaben-Bahn (Lake Constance) operates BR 426 (Bombardier 2-car); BOB Bayerische Oberlandbahn (Munich southeast) operates BR 428 (Stadler FLIRT). The vision rule explicitly pins each. Don't conflate — the new STEP 5 disambiguation rule has the operator pin specifically for this reason.
- **Wikipedia disambiguation pages are a real model failure mode.** When a class number maps to multiple historical machines (e.g. "Baureihe 245" = Saar 1957 V45 OR modern Bombardier TRAXX P160 DE ME 2012, or "Baureihe 428" = pre-war class OR modern Stadler FLIRT), the AI blends them and produces output with mixed-era facts (1930 build year on a 2012 train, etc.). Hardcoded trainSpecs entries are the fix — they short-circuit the AI fallback.
- **Story-arc ads beat product-walkthrough ads by 3-4× on completion rate.** BR 156 / BR 232 / BR 140 v5 (story formats) all clear 17-22% completion. LocoSnap Lifetime announcement (product-walkthrough) hit 5.8%. Pricing-page screenshots at beat 3 are the single biggest scroll-killer. Lifetime as an offer reaches users *after* they install — it doesn't belong in the ad's hook.
- **UK targeting hits an algorithmic ceiling on this DE-anchored TikTok account.** Class 91 v3 with explicit UK theme served 37.6% DE / 27.4% UK because the account's seed audience is German. Diminishing returns on UK-targeting via this account — UK growth path is App Store SEO + word-of-mouth, not TikTok ads.
- **Brand search "Snap loco" emerging at 21% of search traffic on top ads** = first hard unaided-brand-recall signal. Worth tracking as a growth indicator.
- **Play Store conversion 2-5× peer 75th percentile** = real product-market-fit. Whatever Play measures (likely first-open-to-active-user retention), LocoSnap is in the top decile of Travel & Local peers consistently across multiple days.
- **Render auto-deploy is ~3 min from push.** During each deploy window, in-flight `/api/identify` requests can fail with connection errors and trigger Sentry. 4 events / 4 users / 30 days is background noise, not an incident — three deploys today (one per backend ship) explain the recent uptick. If volume jumps to 20+ events in a day, that's a real incident.
- **Sentry "handled: yes" = caught exception, not a crash.** Different severity bucket from a `WatchdogTermination` or `EXC_BAD_ACCESS`. Triage threshold accordingly.
- **CLAUDE.md push-or-hold rule is the right cadence.** Every backend commit got the explicit "Not yet deployed — needs a push to go live on Render. Push now or hold?" prompt before push. User confirmed each. Pattern held.
- **Local commit shows `[claude/priceless-cohen-257fbf SHA]` but pushes via `HEAD:main` land on origin/main correctly.** The 2026-05-09 v1.0.29-not-on-main confusion was a different scenario (worktree branch had real unpushed work that was never merged). This session's pushes were all `git push origin HEAD:main` — verified by `git log origin/main` matching `HEAD`. Future-Stephen note: don't worry about the local branch label.
