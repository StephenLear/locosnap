## What Was Done

### Sentry Build Fix (root cause diagnosed and fixed)
- Identified why `SENTRY_DISABLE_AUTO_UPLOAD=true` in `eas.json` was not working: EAS Build uses fastlane to run Xcode, and fastlane does not propagate shell env vars into Xcode build phase scripts. The sentry-xcode.sh and sentry-xcode-debug-files.sh scripts check `$SENTRY_DISABLE_AUTO_UPLOAD` but the variable was invisible to them.
- Fix: created `frontend/plugins/withSentryDisableUpload.js` — a custom Expo config plugin that injects `SENTRY_DISABLE_AUTO_UPLOAD=true` directly into all Xcode build configurations as an Xcode build setting. Build settings ARE propagated to run-script phases regardless of fastlane.
- Registered plugin in `frontend/app.json` after `"@sentry/react-native"`

## Lessons Learned & Gotchas

- **Fastlane strips shell env vars from Xcode build phases**: EAS env vars in eas.json are available in the macOS shell but NOT in Xcode run-script build phases when launched via fastlane. Only Xcode build settings are reliably visible. Use config plugins to set build settings, not env vars, for anything that Xcode scripts need to read.
- **Check background build tasks before triggering new ones**: Reading the output file first takes 2 seconds and avoids wasting EAS build credits on a duplicate build.
- **Sentry plugin always configures iOS/Android** even when org/project are missing — it falls back to `SENTRY_ORG` env var, which fastlane doesn't expose to Xcode. The plugin is the correct fix, not more env vars.
- **ios/ directory is gitignored** — can't pre-commit sentry.properties there. Config plugins that run at prebuild time are the only way to influence generated Xcode files.
