## What Worked
- Hardcoding specific loco specs in the SPECS_PROMPT (same pattern as ICE 3 family) is reliable for trains the AI is uncertain about
- Extracting frames at 1/3fps to inspect video content before building — saved multiple rebuild cycles once actually used
- Zero-value guard on Wikidata quantity fields is a general pattern worth applying to other fields (power, speed) if similar issues arise

## Key Decisions

- **"Coming soon to Android"** confirmed correct for end screen — Android is beta-only (direct APK to testers), not live on Google Play, so "Now on App Store / Coming soon to Android" is accurate.
- **DR Class 156 pinned in prompt** rather than attempting a Wikidata fix — the Wikidata entity has corrupt/missing data for speed, power and voltage, so pinning is the reliable approach.
- **Zero-weight Wikidata guard** applied broadly (not just DR 156) — any loco with a zero mass claim in Wikidata will now fall through to AI rather than showing "0 tonnes".

---

## Lessons Learned & Gotchas

- **Always read ARCHITECTURE.md spec before building ad assets** — end screen text, font, colours are all specified there.
- **Always run `eas build:list` before drafting tester emails** — never assume the latest version from memory.
- **Extract and READ frames before cutting video** — opening frames in Preview for the user is not sufficient; use the Read tool to inspect them directly.
- **Tester email purpose** — the ask is Google Play opt-in (contributes to Play Store launch requirements), not in-app account registration.
- **Wikidata quantity claims can have amount = 0** — this is a data quality issue, not a missing claim. Guards must check both existence and non-zero value.
