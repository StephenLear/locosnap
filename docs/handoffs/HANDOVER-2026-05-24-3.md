### Backend — VR Sr1 facts-layer wholesale lock (`de1e3c6`)

Finnish TikTok commenter "Deevee" (Finnish flag emojis in handle) posted a screenshot of a VR Sr1 scan captioned "What is this 😭". Specifications panel was entirely correct — 160 km/h / 3,100 kW / 84 tonnes / Novocherkassk / Finnish broad gauge / Electric 25 kV 50 Hz — all sourced from `KNOWN_SPECS`. The hallucination was confined to the facts-layer prose:

> *"The VR Sr1 was Finland's first mainline electric locomotive — a groundbreaking 1920s Bo-Bo..."*
> *"The Sr1 was one of the earliest mainline electric locomotives in Scandinavia, entering service in the 1920s..."*

Two wrong claims: (1) **Decade.** Sr1 prototype 3001 was delivered 1973, production ran 1973–1985 with a small later batch through ~1996. Finland did not electrify its mainline network until 1969 (Helsinki–Kirkkonummi was the first electrified section), so the 1920s framing is geographically and historically impossible. The "Finland's first mainline electric" framing IS correct in context but the decade is fabricated by ~50 years. (2) **Wheel arrangement.** Sr1 is Co'Co' (six axles, two three-axle bogies), not Bo-Bo. The trainSpecs.ts vision-prompt block explicitly forbade Bo'Bo' for Sr1, but trainFacts.ts had no Sr1 bullet — facts narrative was free-form and the LLM padded with plausible-sounding wrong claims.

**Same root pattern as yesterday's ÖBB 4020 wholesale fix.** Specs panel held because KNOWN_SPECS pinned typed fields; facts narrative drifted because no class-specific lock existed. Second class in 24 hours with this failure mode (ÖBB 4020 yesterday, VR Sr1 today). Pattern now firmly established as the dominant facts-layer leak — any class with a KNOWN_SPECS entry but no `trainFacts.ts` bullet is at risk.

**Fix shipped:**
- `backend/src/services/trainFacts.ts` — new comprehensive VR Sr1 bullet inserted above the existing VR Dv12 bullet (keeps Finnish content adjacent). Hard-locks: (a) entry year **1973** — forbids "1920s", "1930s", any pre-1970 date, any "entering service in the 1920s" framing; (b) wheel arrangement **Co'Co'** — forbids "Bo-Bo", "Bo'Bo'", any four-axle attribution; (c) builder **Novocherkassk (NEVZ) / Strömberg** joint Soviet-Finnish project — forbids Siemens / ABB / Adtranz / Bombardier / SLM / Stadler / any Western European builder; (d) operator **VR only** — Sr1 has never been exported or operated outside Finland (broad-gauge spec alone restricts it); (e) fleet size **110 units**; (f) current status framed as "Finland's first mainline electric workhorse from the 1970s through the 2010s, now gradually being replaced by Sr3 Vectron from 2017 onwards" — forbids "extinct" / "all withdrawn" / "preserved only" / "museum class" / "phased out completely". ## What Didn't Work (and Fixes)

- **Translation-alongside rule dropped on the second-pass draft.** First-pass prepaid-email draft included English translations under each German and Polish body. Second pass (substituting real recipient names) showed only the foreign-language bodies and dropped the translations. User pushed back: *"what am i sending as i dont speak polish or german - this is not the first time i have had to mention this"*. This is a repeat failure mode the user has had to correct before. **Root cause:** the existing memory rule (`feedback_translation_alongside_drafts.md`) was written as if the first draft was the only one — no explicit handling for iteration, name substitution, or "final version" presentation. **Fix:** appended a new section to the memory rule covering iteration explicitly: every revision must carry the translation, applies to subject lines as well as bodies, with an explicit pre-send check the agent must run on any message containing foreign-language text. Treat draft and translation as a single inseparable unit.
- **Initial "lose all their spots" framing was inaccurate.** User's framing carried the implication that Pro expiry would delete the spots. Quick grep on `FREE_COLLECTION_LIMIT` confirmed the actual gating: only most-recent 5 spots shown without Pro, the rest are locked but persist in Supabase. **Fix:** corrected the framing in the email bodies to "your collection stays safe but most of it will be locked unless you re-subscribe" — materially less alarmist and accurate. Caught before any draft went out.

## Key Decisions

- **Use email rather than in-app banner for the prepaid recovery outreach.** PL user expires in 4 days, DE user in 5 days. An in-app banner would need a frontend code change → EAS build → Apple/Google review → user opens the app in time. Even with the fastest dual-store turnaround this session has measured (~24h), that's 3 days for the banner to surface — and they may not open the app in time. Email lands today and works regardless of app re-open. For **future** prepaid users, an in-app banner IS the right long-term solution; logged for v1.0.35.
- **Single-paragraph "this was our setup mistake" disclosure rather than vague service language.** Reasoning: customers respond better to owned-and-explained mistakes than to "we noticed your subscription needs attention" euphemism. Lower comeback rate, fewer follow-up questions, and protects credibility if the same customer purchases again post-fix.
- **Don't offer a free Pro grant to compensate.** Per `policy_pro_grants_discontinued.md` (2026-05-17), Pro grants are discontinued. This is borderline — the policy was about creator/tester grants, not service-recovery grants — but a grant would (a) not solve the actual problem (they still need an active subscription to keep Pro after the grant expires), and (b) train the response that "complaint = free Pro" which the policy explicitly prevents. The email instead points to the correct re-subscribe path that now goes to auto-renewing.
- **Ship the Sr1 facts-layer fix proactively across all KNOWN_SPECS variants rather than the minimum required.** Yesterday's morning ÖBB 4020 stage-3 fix had to retrofit missing variants after a "still wrong" comment. Today's Sr1 fix included every plausible Vision class-string variant from the first commit — the cost is essentially free (7 vs 2 entries in a string-map), the cost of NOT doing it is another public correction loop on a different commenter. First proactive application of the checklist established this morning.

## Lessons Learned & Gotchas

- **Translation rule must persist across every iteration of a draft, not just the first version.** Codified in the memory update. The specific failure modes documented: name-substitution passes, wording tweaks, "final version" presentations, subject-line presentations. The pre-send check the agent now runs: "is there a complete English translation of the latest version in this same message?" If no, the message is incomplete. Treat draft and translation as a single unit.
- **Facts-layer hallucinations now confirmed as a class-wide pattern, not class-specific.** Two unrelated classes in 24 hours (ÖBB 4020, VR Sr1) failed the same way: KNOWN_SPECS pinned the specs panel correctly, no `trainFacts.ts` bullet meant the facts narrative was free-form, LLM padded with plausible-but-wrong claims. Every class with a KNOWN_SPECS entry but no `trainFacts.ts` bullet is at risk. A one-time audit of the KNOWN_SPECS keys vs the `trainFacts.ts` bullet list would surface every at-risk class — flag for the next quiet-content session.
- **The variant-coverage checklist pays back on first use.** Yesterday's ÖBB 4020 stage-3 fix added the checklist as a memory rule after a retrofit was needed. Today's Sr1 fix applied the checklist proactively and required no retrofit. Both data points support keeping the checklist as canonical going forward.
- **Resend returns 200+ID on suppressed addresses.** Already in memory (`feedback_resend_suppression_silent.md`) but worth re-flagging on every batch: API-acceptance ≠ delivered. The two prepaid recovery emails are accepted but `last_event` must be verified in the Resend dashboard before declaring the recovery loop closed. Suppression isn't expected here (neither email appears in prior welcome batches and there's no obvious bounce signal), but cheap to check.
- **The "Pro grant" reflex needs to be checked against `policy_pro_grants_discontinued.md` every time.** When a paying customer's subscription is at risk, the easy intuitive answer is "comp them with a free month". Per policy this is now off the table. The discipline is to find an actual fix in the product / subscription flow rather than papering over with a comp.
