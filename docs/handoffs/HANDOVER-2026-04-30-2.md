## What Worked
- **TDD task ordering of the plan.** Each task wrote the failing test first, then the implementation. Caught the `services/supabase.ts` mock pattern issue early (jest.setup.ts mocks aren't auto-loaded — confirmed inline mocks needed in new test files).
- **Independent reviewer agents.** Three passes caught issues self-review missed. The `superpowers:code-reviewer` agent is most useful when given an explicit checklist (pass 3's prompt was ~700 words and surfaced the cleanest residual set).
- **Convention-based i18n keys for emoji labels.** `t(`spotterEmojis.${id}`, { defaultValue: emoji.label })` — no need to add a `labelKey` field to every entry; falls back to English on missing keys.
- **`Intl.DateTimeFormat().resolvedOptions().locale` for device locale.** Avoided adding `expo-localization` dep; works in Hermes; verified `pl-PL` → region `PL` → resolves to Poland.
- **Wrapping migration 011 in BEGIN/COMMIT.** DDL invisible to other sessions until commit. Combined with the `RAISE EXCEPTION` 010-dependency guard, the migration is now safe to apply on a live DB.

## What Didn't Work (and Fixes)

| Problem | Root cause | Fix |
|---|---|---|
| Onboarding email step caused duplicate OTP send | `/sign-in` ignored `email` param, user retyped, second OTP issued, first invalidated | `/sign-in` now reads `email` + `autoSend`; onboarding routes via these params with `markIdentityOnboardingComplete()` called first |
| Re-prompted onboarding after OTP verify | `markIdentityOnboardingComplete` only set AsyncStorage flag for anonymous users; server `has_completed_identity_onboarding` stayed false | `migrateAnonymousIdentity` now also lifts the AsyncStorage onboarding flag onto the server profile during `fetchProfile` |
| AsyncStorage anon-identity values leaked forever | When server already had both fields populated (e.g. signed in on another device), the helper returned null without clearing AsyncStorage | Helper now clears stale anon country/emoji keys whenever server has both fields populated |
| 16 Pro SVG emojis would have shipped as broken letter-tiles | SVG asset bundle hasn't been sourced yet | Filter `SPOTTER_EMOJIS` to Unicode-only via `.filter(e => e.source === 'unicode')`. Full set retained as `ALL_SPOTTER_EMOJIS` so legacy values stored on the server still resolve via `getEmojiById` when the assets land |
| Onboarding gate stale-timer race | `clearTimeout` cleanup was returned from inside `.then()`, never reached the useEffect cleanup path | Lifted `timeoutId` ref + `cancelled` flag both into the outer cleanup |
| Migration 011 could expose "relation does not exist" mid-DROP | DROP VIEW takes ACCESS EXCLUSIVE lock; non-transactional, in-flight selects could fail | Wrapped DROP+CREATE in `BEGIN; ... COMMIT;` |
| German users would have seen "Leaderboard" mid-flow when rest of app says "Bestenliste" | Author drafted DE strings without checking existing translations | Reviewed `locales/de.json` for prior usage; updated to "Bestenliste" |

## Key Decisions

- **Branch strategy:** `feat/leaderboard-phase1` rather than direct-to-main commits per the plan's literal instructions. User confirmed "branch feels safer" early in session. 26 commits on the branch, none pushed to origin.
- **Skipped `expo-localization` dep:** Used existing `settingsStore.language` + `Intl.DateTimeFormat()` for locale detection. No new dependency added. Verified PL resolution works.
- **Added migration 011 beyond plan:** The plan only specified 010 (column add). But the leaderboard views were defined as `SELECT p.id, p.username, p.avatar_url, p.level, ...` with explicit column lists, so they don't auto-pick up new profile columns. 011 was necessary.
- **Email step routes to `/sign-in` not embedded OTP:** Reuses existing OTP UX rather than reimplementing in onboarding. Cleaner. Required adding `email` + `autoSend` params to `/sign-in`.
- **SVG Pro emojis filtered out for v1.0.22:** Better than shipping placeholder letter tiles to paying Pro users. Add SVG assets in v1.0.23 then drop the filter.
- **Onboarding flag retained in AsyncStorage even after migration:** The gate also reads it for unauthenticated users, who may have a session-less period after sign-out. Country/emoji keys cleared post-migration; onboarding flag deliberately preserved.

## Lessons Learned & Gotchas

- **`jest.setup.ts` mocks are NOT auto-loaded.** Confirmed: `jest.config.js` has no `setupFilesAfterEach` referencing it. Each test file must inline its own mocks (analytics, purchases, supabase config). Pattern from `__tests__/store/trainStore.test.ts` works.
- **`services/supabase.ts` imports `expo-file-system/legacy` (ESM).** Test files that exercise the real module must `jest.mock('expo-file-system/legacy', ...)` and `jest.mock('base64-arraybuffer', ...)` to get past Jest's ESM parser.
- **`config/supabase` vs `services/supabase` are different files.** `config/supabase` exports the supabase client; `services/supabase` exports CRUD helpers. `_layout.tsx:16` has a pre-existing tsc error from importing `supabase` from `services/supabase` (which doesn't export it). Runtime works via bundler shenanigans. Worth fixing eventually.
- **Iterative review pattern is a session-cost.** First-pass review missed obvious cross-file i18n leaks. Second-pass review missed a few NICE-TO-HAVE items. The user is right that a single thorough first-pass is the bar. Memory file `feedback_check_completeness_first_time.md` codifies the checklist for next session: i18n parity / no-emoji glyphs / dead code / cross-file consistency / test gaps before asking for review.
- **The "no emojis" CLAUDE.md rule applies to files (output/communication), not to legitimately data-typed in-app iconography.** Country flags in `data/countries.ts` and spotter emojis in `data/spotterEmojis.ts` are app data, not output. Confirmed by the third reviewer. The 🔒 lock indicator was the wrong call — that was decoration, not data — swapped to `Ionicons name="lock-closed"`.
- **CREATE OR REPLACE VIEW can't reorder columns.** That's why migration 005 used DROP+CREATE. For 011, kept the same DROP+CREATE pattern but wrapped in a transaction for atomicity, since changing column order is essentially required when adding columns into the middle of an existing column list.
- **Hermes Intl support works for DateTimeFormat.** Tested mentally; verified via Node behaviour. `Intl.DateTimeFormat().resolvedOptions().locale` returns the device locale string. Wrapped in try/catch as a defensive measure for older Hermes builds.
