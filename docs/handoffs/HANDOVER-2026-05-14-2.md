## What Was Done (this delta)

### Supabase Data API grant convention — full convention update
- **Trigger:** Supabase change-notice email 2026-05-14 evening. Default `public`-schema grants on the Data API are being removed: 2026-05-30 for new projects, 2026-10-30 for all existing projects (including LocoSnap).
- **Impact audit:**
  - LocoSnap is affected — frontend (`frontend/config/supabase.ts` createClient with anon key) and backend (`backend/src/config/supabase.ts` createClient with service_role key) both use supabase-js = Data API. No direct Postgres connection string anywhere.
  - **Zero existing GRANT statements** across all 15 migrations (`supabase/migrations/001-015.sql`). Pattern is `CREATE TABLE → ENABLE RLS → CREATE POLICY` with no GRANT step.
  - Existing 13 tables retain their current grants permanently per the email. Risk is forward-only.
- **Convention updates (all committed `a2457b9`):**
  - `CLAUDE.md` — new "Supabase migration template" mandatory rule with full SQL template (`grant select on public.x to anon`; `grant select, insert, update, delete on public.x to authenticated`; same for `service_role`; `enable row level security`; at least one policy). Per-table tightening guidance: read-only tables drop CRUD from clients; backend-only tables omit grants entirely; `service_role` keeps full CRUD always. Pre-merge verification steps included.
  - `docs/ARCHITECTURE.md` — new "Data API grants — convention change effective 2026-10-30" subsection under "4. Database — Supabase" with same template and pre-cutover Security Advisor check note.

## Key Decisions

- **Convention rule in CLAUDE.md, not just memory.** Memory entries can be skipped if forgotten; CLAUDE.md rules are loaded into every session automatically. The Supabase grant pattern is structural enough that a CLAUDE.md rule is the right level. Memory entry is a secondary cross-reference.
- **Don't backfill migrations 001-015.** Idempotent + no functional effect + risk of introducing a mistake in mechanical edits. Pattern-match argument for future authors is real but weak — they'll see the CLAUDE.md rule first.
- **Per-table tightening guidance included in the template.** Read-only tables (lookup data) don't need CRUD for `authenticated` / `anon`. Backend-only tables don't need any grants. Default template is the most-permissive case; tighten where appropriate. Without this guidance the rule could lead to over-broad grants on tables that shouldn't be CRUD-able from clients.
- **Pre-cutover Security Advisor check.** Single low-cost ops step before 2026-10-30 to verify no warnings. No need for a calendar reminder — the rule is in CLAUDE.md and the advisor is the safety net.

## Lessons Learned & Gotchas

- **The new Supabase rule is forward-only, not retroactive.** Existing 13 tables are safe forever. Only NEW migrations after the cutover need the new template. Easy to misread the email as "you need to fix everything" — the load-bearing sentence is "Existing tables keep their current grants."
- **Backend supabase-js is also Data API, not direct Postgres.** Don't assume the backend's service_role key gives it a different code path. `createClient` from `@supabase/supabase-js` is Data API regardless of which key is used. If the backend ever needs to bypass this for performance or guaranteed access, that requires `pg` / `postgres` / similar direct-Postgres library — none currently in use.
- **Audit migrations with `grep -c "grant\|GRANT"` to verify convention.** Zero result confirmed every existing migration relies on defaults. This is fast and answers the "are we already doing this?" question definitively.
