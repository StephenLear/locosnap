## Lessons Learned & Gotchas

- **`WIKIDATA_CORRECTIONS` only allows six fields**: `maxSpeed | power | weight | builder | fuelType | numberBuilt`. Length, gauge, status, route, length all must be enforced via the system prompt instead. TypeScript will fail the build on any extra field.
- **Source clips from enthusiast creators almost always carry a watermark.** AELP on Das war's, "Beautiful sound" on the Eurostar clip for the launch video, various rail-channel badges on others. Check before building, cover with a black bar if present — positioned above the overlay text so it reads as intentional framing.
- **Class-number collisions are a category of bug, not a one-off.** Already hit on BR 232, BR 648, Class 52, BR 442, BR 151. Pattern: train visually identified correctly by vision, but class-name string is ambiguous (just "Class X" or just "BR X" without operator prefix) and the specs layer hits the wrong database entry. Fix is always: (a) force vision to return an operator-prefixed class string, (b) hardcode the correct class variant in specs prompt + WIKIDATA_CORRECTIONS.
- **Rarity layer derives cleanly from corrected specs.** For BR 232, BR 648, and now BR 151, fixing the specs alone made rarity return the right answer without a separate rarity override. Only touch `rarity.ts` if specs-based rarity still drifts.
- **Beat 1 and beat 5 timestamp errors are the two most common cut-mistakes.** Beat 1 often starts too early (train not in frame yet); beat 5 often starts too early (Identifying phase instead of card reveal). Always extract a verification frame from each beat before concat.
- **Render auto-deploy on push** is reliable (verified via re-scan returning correct data ~3 min after push). Don't need to explicitly trigger.
