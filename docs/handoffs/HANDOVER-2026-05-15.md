## What Worked

- **Brainstorming skill ran cleanly.** Four questions, AskUserQuestion tool, multiple-choice answers. Each answer shaped the design without backtracking. The hardgate ("present design + get approval before writing code") forced the second-review pass that caught the `default`-variant null-bailout misstep before any code was written.
- **Second-review catch saved a regression.** Initial design proposed returning `null` from `PaywallSoftPrompt` when variant is `default`. Mid-implementation I realised this would silently remove the generic "Grow your collection" banner from the results screen at scans 0/1/3/6+ — that copy IS the existing live UX, not a stale code path. Revised before coding: keep `default` on results screen, gate it OUT only on the camera-surface mount via the `[2,4,5].includes(...)` wrapper check. No regression shipped.
- **Reusing existing component instead of building new.** ~30 LOC end-to-end because `PaywallSoftPrompt` already had urgency styling, dismiss state, analytics, route routing. The `surface` prop + variant-reset effect were the only behavioural additions.
- **`surface` prop with sensible default.** `surface = "results"` keeps results.tsx call site non-breaking. Zero changes required to `app/results.tsx`. Clean prop-design pattern for adding a parallel mount.
- **File-based commit message via `/tmp/commit_msg.txt` + `git commit -F`.** Heredoc would have broken on the `$rc_lifetime`-style dollar signs and `daily_scans_used ∈ {2, 4, 5}` Unicode in the message body. First-try success per the rule. Cleaned up `/tmp/commit_msg.txt` after.
- **Architecture doc gained behavioural symmetry note.** The new paragraph explicitly notes "default variant fires on results screen at scans 0/1/3/6+; camera surface gates it out" — important asymmetry that future sessions need to understand without re-reading code.

## Key Decisions

- **Full variant ladder (scan_2/4/5) on camera, not just scan_5.** User explicitly chose this over the narrower "scan_5 only" interpretation of the backlog memo. Consistency with results-screen behaviour; more touch-points for measuring whether the new surface moves conversion.
- **Dismiss resets on variant escalation.** Critical for the persistent camera tab. Without it, a dismiss at scan 2 would permanently silence the scan_5 amber banner — defeats the entire purpose of the new surface.
- **`default` variant gated OUT on camera surface, retained on results.** Asymmetric on purpose. Camera surface only paints when there's specific urgency copy; results surface keeps the generic encouragement banner for non-Pro users between escalation points.
- **Pre-signup trial users see nothing on new surface.** They keep their existing trial banner. Adding a 4th variant ladder for the 0..3 trial domain would have ballooned scope and required new i18n copy. Deferred — re-open if early data shows trial users converting below benchmark.
- **`surface` prop in analytics + paywall route source param.** Telemetry critical to read v1.0.32 — without it we couldn't tell whether camera or results banner is driving tapped/conversion events. RC source param means purchase attribution can also split by surface.

## Lessons Learned & Gotchas

- **Read the i18n keys before claiming a code path is stale.** I almost shipped a regression by assuming `paywall.softPrompt.default` was dead code. It isn't — it's the load-bearing copy for the generic results-screen banner that fires at most scan counts. Process: when proposing to remove or null-out a branch, grep for the keys that branch depends on and inspect every caller before claiming the path is unused.
- **Worktree node_modules drift is a known pattern.** 2026-05-14 handover already flagged this. Today repeated the symptom — couldn't tsc from worktree. Either commit to installing node_modules in worktrees once-per-checkout, or accept that worktree TS work goes uncompiled until pushed back to main and tested there.
- **`surface` propification pattern is reusable.** Any time we want to mount the same component on two screens with slightly different copy/analytics/route gates, an optional `surface` prop with a sensible default is the cleanest implementation — non-breaking for existing callers, no duplicated component, no shared state between mounts.
- **Brainstorming skill HARD-GATE is real.** It forced one extra exchange (design presentation + approval) that initially felt like overhead but caught the `default`-variant regression. Treat it as a feature, not a tax.
- **AskUserQuestion is the right tool for multi-choice scoping questions.** Faster than freeform back-and-forth; captures the decision verbatim in the conversation history.
- **The variant-reset useEffect is subtle but load-bearing.** It only matters on a persistent surface (the camera tab). Without it, the results-screen behaviour is unchanged because each result is a fresh component mount. Comment in the source code explains the asymmetry so a future maintainer doesn't strip it as "unused state reset".
