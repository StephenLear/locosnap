## Lessons Learned & Gotchas

- **Wikidata can map to catastrophically wrong entities.** BR 483/484 was returning Crewe Works / 1943 / 15kV AC — a completely different heritage loco. Always verify specs against the class's known production period and voltage system before trusting scraped data. When a tester reports "specs wildly wrong," suspect Wikidata entity mis-match first, not prompt hallucination.
- **Monaco editor content is not saved on `setValue()`.** Replacing editor content via `monaco.editor.getEditors()[0].setValue(...)` only modifies the in-memory buffer. Saved queries on disk stay intact unless user clicks Save. Safe for ad-hoc migrations.
- **When results contain emails/PII, data extraction blocks.** Claude-in-Chrome returned `[BLOCKED: Cookie/query string data]` when I tried to read a SELECT output containing email addresses. **Workaround:** rewrite queries to return aggregate/derived columns (counts, pattern-matches) rather than raw email values.
- **For TikTok closer-beat text, 11-char max.** "AND COUNTING" (12 chars) overflowed at 64pt on a 720-wide frame. Use "STILL GOING" / "STILL HERE" / "66 YEARS" as safe alternates.
- **Czech tester "Coladose" nickname for BR 485.** Worth remembering for any future German S-Bahn video — the DDR-era trains have Coca-Cola-bottle nicknames that carry real emotional weight in enthusiast circles.
- **Tom (Lxcx_241) is a premier Czech rail feedback source.** Six separate issues in one batch, all legitimate. Keep his DM warm — he's the reason two rarity tiers got corrected and three disambiguation rules got added.
- **The app falls back to `email.split("@")[0]` when `profile.username` is null.** If usernames look like email prefixes post-login, it's a cached-state issue, not a DB issue. Force-close the app.
