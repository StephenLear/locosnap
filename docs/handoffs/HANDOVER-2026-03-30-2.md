## What Worked

- **Statistical tiebreaker** — the decisive unlock for BR 412. Visual feature descriptions alone couldn't overcome Claude's default priors. Telling the model that BR 412 is 6x more common than BR 407 at major German stations flipped the result. Lesson: for ambiguous visual classes, frequency priors are as important as feature descriptions.
- **Moving ICE rules to top of prompt** — burying the "ICE 3 is forbidden" rule in the disambiguation section (50+ lines in) meant the model had already committed by the time it read it. Putting it as a PRE-FLIGHT CHECK at the very top worked.
- **Claude Vision over GPT-4o** — Claude correctly identified the ICE 4 in direct testing on the first attempt. GPT-4o never got it right despite many prompt iterations. Infrastructure was the real blocker, not the prompt.

## What Didn't Work (and Fixes)

- **Generic "ICE 3" ban alone** → Claude returned "ICE 3" even with an explicit "FORBIDDEN" label. Fix: moved the ban to the top of the prompt AND added the statistical tiebreaker together.
- **BR 407 at 88% confidence** → after banning "ICE 3", Claude landed on BR 407 instead. Fix: added chin edge discriminator + 6x BR 412 frequency tiebreaker.
- **WIKIDATA_CORRECTIONS not firing for max speed** → turned out the user misread 250 as 230 (typo). Corrections were working. Added extra class string variants anyway as defensive measure.
- **Old API key accidentally shared in chat** → user pasted a `sk-ant-...` key directly into the conversation. Key was revoked immediately. No harm done but worth noting: always go straight to Render's Environment tab, never paste keys into chat.

## Lessons Learned & Gotchas

- **Prompt position matters as much as content**: The same rule buried at line 70 vs line 1 produces different model behaviour. Critical constraints belong at the top of the prompt, not in a disambiguation appendix.
- **Cache version bumps are essential during iterative fixes**: Every time a wrong identification gets stored in Redis (e.g. "ice 3::db"), it will serve that wrong result for 30 days regardless of prompt changes. Bump the version whenever the model was returning wrong data that could have been cached.
- **Claude Vision and GPT-4o have different priors**: Switching models can resolve identification issues that prompt engineering cannot. Test both providers when a class is consistently misidentified.
- **The screen recording for the ICE 4 ad shows "Spotted x2"** — the card reveal in `ScreenRecording_03-30-2026 22-44-11_1.mp4` shows multiple scans. Not ideal but usable. A clean "New class added" first-scan recording would be better for future ads.
