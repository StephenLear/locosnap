## What Worked

- **Word-for-word match diagnosis.** Reading the third screenshot carefully and noticing the prose matched the pre-fix hallucinations verbatim — letter-for-letter, not paraphrased — was the key insight. If it had been the LLM ignoring the system-prompt update, the wording would have varied. The verbatim match could only mean a stale-cache hit. That narrowed the investigation immediately to "why didn't the invalidation map fire" rather than "is the prompt update strong enough".
- **Cross-referencing KNOWN_SPECS keys vs CLASS_INVALIDATIONS keys.** Once the diagnosis pointed at cache invalidation, the natural next move was to enumerate every variant `trainSpecs.ts` KNOWN_SPECS covers and check each one against `CLASS_INVALIDATIONS`. The 2-key gap (öbb baureihe 4020, reihe 4020) jumped out immediately.
- **Defence-in-depth timestamp bump.** Bumping the invalidation timestamp from `2026-05-23T22:00:00Z` to `2026-05-24T07:50:00Z` was cheap insurance — even if the diagnosis was right that the gap was the primary cause, any entry cached between yesterday's fix and this morning under an uncovered variant gets caught too. Belt-and-braces.
- **`/api/health` `cache.totalEntries` as a deploy-confirmation signal.** No version field, no commit hash on the health endpoint, but `totalEntries` resetting from 38 to 0 unambiguously indicated the service restarted (in-memory L1 cleared). Good enough confirmation that Render rolled out the new build without needing dashboard access.
- **Asking before shipping.** Used `AskUserQuestion` once before the cache-key fix to confirm the user wanted to ship vs investigate vs ask user to rescan first. Recommended option was approved and shipped first try. Better than the equally-cheap alternative of "ship, then ask if it was the right call".

## What Didn't Work (and Fixes)

- **Yesterday's wholesale fix was scoped to known-failing keys only.** The 2026-05-23 fix added entries for the 4 KNOWN_SPECS variants that Vision had been observed returning at that point, not all 6 KNOWN_SPECS variants. Worked for the specific scans that triggered the original complaint; failed for this morning's scan that hit a previously-untested variant. Fix: the new variant-coverage rule in `backend_cache_invalidation_pattern.md` requires mirroring every KNOWN_SPECS key plus common English/ASCII variants on EVERY future per-class fix, removing this whole failure mode.
- **Background poll for "deploy ready" was the wrong tool.** First attempt polled `/api/health | grep -q '"status":"ok"'` to detect deploy completion. That returns immediately because the OLD service was already healthy — useless as a "new build live" signal. The correct check was `cache.totalEntries` resetting to 0 (in-memory L1 cleared on restart). Caught after one wasted poll cycle.

## Lessons Learned & Gotchas

- **Word-for-word matches between bug reports and historical wrong output = cache hit, not fresh hallucination.** If the LLM is failing on the same class twice, varying wording means the model is re-hallucinating (prompt fix needed); verbatim wording means a stale cache layer is leaking through (invalidation fix needed). This is a fast diagnostic — five seconds of careful reading vs an hour of prompt-vs-cache hypothesis testing.
- **Every `KNOWN_SPECS` lookup key needs a matching `CLASS_INVALIDATIONS` entry.** Now codified in `backend_cache_invalidation_pattern.md` with a 4-step checklist. The next person to add a `CLASS_INVALIDATIONS` block (likely the next per-class correction) should follow the checklist mechanically rather than copy yesterday's pattern, which would have re-introduced this exact gap.
- **`/api/health` does NOT carry a build-version field.** `version: "1.0.0"` is hardcoded and doesn't change between deploys. Use `cache.totalEntries` (resets to 0 on restart) as the "new build live" signal, OR poll the Render dashboard. If we ever care about this often, adding a `buildSha` field to the health endpoint sourced from a `GIT_SHA` env var at build time would close the gap cheaply.
- **Polling `until curl ... | grep -q '"status":"ok"'` is the WRONG pattern for deploy-detection** when the old service is already returning OK. The first call succeeds against the old build, the loop exits, you think the new build is live. Pick a signal that genuinely differs between old and new builds (cache stats, env-vars, fingerprint).
- **Memory files referenced in MEMORY.md but living outside the repo can't be committed alongside code changes** — `~/.claude/projects/.../memory/*.md` files are tracked separately. When updating both memory and code in the same session, two commits / two stagings are required (or use two `git add` commands with the absolute paths from different repos). Caught it gracefully today; future sessions should expect this.
- **Three-stage correction loops on a single commenter are recoverable IF each stage ships fast.** The same DACH commenter has now produced three rounds of feedback on the same class within 24 hours. Each round had a same-session backend fix and a same-thread public reply. Credibility loss across three stages would have been significant if any single round had taken >24h to land. The "ship same session" reflex is non-negotiable for live-correction threads.
