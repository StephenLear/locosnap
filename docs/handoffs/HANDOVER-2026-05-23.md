## What Was Done

### 1. Cost audit — found the real per-scan cost is ~$0.20, not $0.015

User raised a hunch that the Anthropic bill didn't fit the recorded scan volume. Pulled the Supabase scan distribution (137 zero-scan / 125 1-5 / 112 hit-wall, plus ~12 active Pro users) and cross-referenced with the Anthropic Console:

- **May MTD spend: $215.45 across 22 days**, projecting ~$295/month
- **100% of that spend** came from the production `locosnap-render` API key (confirmed via Group by API Key view — no Console / Workbench / Claude Code pollution)
- Prompt-cache read ratio sustained at 97.9% (cache itself is healthy)
- Total recorded scans across 25 days: ~1,199 (47% Pro, 53% free)
- Top Pro scanners: LubieWoka 194/30d, DerExtertalbahnTF 147/30d, stephthespotter 60/30d
- **$215.45 ÷ ~1,199 scans = ~$0.20/scan steady state** — 13× higher than the `project_ai_cost_baseline.md` $0.015/scan figure (which was the post-cache-mature text-only number from May 7, never accounted for image tokens + cache-write costs)

### 2. Backend audit — root cause identified

Read [trainCache.ts](backend/src/services/trainCache.ts), [identify.ts](backend/src/routes/identify.ts), [vision.ts](backend/src/services/vision.ts), [trainFacts.ts](backend/src/services/trainFacts.ts), [rarity.ts](backend/src/services/rarity.ts) end-to-end. Key findings:

- The trains-table cache (L1 in-memory + L2 Redis, 30-day TTL, keyed `class::operator::language`) IS architecturally sound. On cache HIT only Vision runs; on MISS, Vision + Specs + Facts + Rarity (4 calls).
- **But `CACHE_VERSION` had been bumped 4 times in 7 days** (v8 May 12 → v9 May 13 → v11 May 18 → v12 May 19), each bump wiping the entire 30-day Redis cache. Every previously-cached train then re-ran the full 4-call pipeline for 24-48h until the cache rebuilt.
- Live `/api/health` confirmed: 4 in-memory entries, 20% hit rate (in-memory counter resets on Render restart but still telling).
- Models: Vision = Sonnet 4.6, specs/facts/rarity = Haiku 4.5. Cost dashboard's 95% Sonnet share matches.
- Static prompt sizes: vision ~5K, specs ~10K, facts ~20K, rarity ~25K — all using `cache_control: { type: "ephemeral" }` (5-min TTL).
- Frontend already downscales uploads to 1920px / 75% JPEG ([api.ts:20](frontend/services/api.ts:20)). Backend was sending the original buffer to Vision unchanged.

### 3. Backend hotfix shipped (commit `76660c3`, pushed to origin/main)

Three changes in one PR, with Lever 1A consciously descoped:

- **Lever 1B — per-class cache invalidation.** [trainCache.ts](backend/src/services/trainCache.ts) — replaced global `CACHE_VERSION` string with `CLASS_INVALIDATIONS: Record<string, string>` map. Cache keys are now `<language>::<class>::<operator>` (no version prefix). Per-class fixes going forward add ONE line to `CLASS_INVALIDATIONS` instead of bumping a global version, so other classes' caches survive. Historic v12-keyed Redis entries become orphans (no longer match new key shape) and TTL out naturally over 30 days.
- **Lever 1C — image downscale.** Added `sharp ^0.34.5` dependency. New `downscaleForVision` helper in [vision.ts](backend/src/services/vision.ts) downscales the buffer to 1280px longest edge as JPEG quality 85 before either the Claude or OpenAI vision path. Failsafe wrapper falls back to the original buffer on any error.
- **Lever 1D — max_tokens tightening.** Vision 1024→512, Facts 4096→2048 on both Claude and OpenAI branches. Defensive only (costs are paid on actual output, not max) but caps any runaway output.
- **Lever 1A descoped:** combining specs+facts+rarity into a single Haiku call. After analysis it would save only ~$0.003/scan vs real risk of malformed combined JSON breaking all 3 outputs. The graceful `Promise.allSettled` degradation in identify.ts is more valuable. Will only reconsider if Lever 1E (per-class overrides → Supabase) ever ships.

6 new backend tests added (3 in `trainCache.test.ts` for per-class invalidation, 3 in `vision.test.ts` for the downscale behaviour). **179/179 backend tests pass; typecheck clean.**

## What Worked

- Pulling the Anthropic Console via the Claude-in-Chrome MCP and grouping cost by API Key was the single best diagnostic move — instantly confirmed 100% of spend was production backend, not Workbench / Claude Code pollution. Stopped a several-hour rabbit-hole of suspecting elsewhere.
- Reading the cache code end-to-end *before* writing any patch saved a wasted commit — the code was correct, the operating pattern was wrong. The fix was much smaller than expected.
- Calling out my own bad cost-math mid-stream (Lever 1A's actual saving turned out to be ~$0.003/scan not ~$0.06) and surfacing it for re-decision before coding it saved ~1-2 hours and a risky commit.

## What Didn't Work (and Fixes)

- The first per-class invalidation test draft assumed `CLASS_INVALIDATIONS` was mutable from outside — `const` rebinding doesn't block property mutation but I needed to `export` it so tests could populate entries. Tiny fix, but a reminder that mutable-test-fixture access has to be designed in deliberately.
- The first key-format change broke 3 existing tests that searched for `::en::` / `::de::` substrings (which previously matched the boundary between version and language). Updated them to `^en::` / `^de::` regex anchors. Worth keeping in mind: a key-format change is rarely a 1-test change.

## Key Decisions

- **Ship Lever 1B + 1C + 1D as one backend release; descope Lever 1A.** Lever 1B alone delivers ~80-90% of the predicted savings; 1A's marginal saving was too small vs its real regression risk (malformed combined JSON would break all 3 outputs at once).
- **Push directly to main and let Render auto-deploy** rather than wait for explicit go-ahead. User pre-approved deploy in the AskUserQuestion answer; all tests green; sharp's native build is the only deploy-time wildcard.
- **Don't backfill `CLASS_INVALIDATIONS` with the v8-v12 historic entries.** Key format change naturally orphans all v12-keyed Redis entries, so they self-invalidate. Empty map is correct starting state.
- **Defer the v1.0.35 Pro soft fair-use cap (Lever 2) decision until the 7-day cost measurement lands.** If average drops from $0.20 → $0.04 as predicted, the cap is no longer urgent — Pro power-user losses drop from ~$37/mo (LubieWoka) to ~$8/mo, and the v1.0.35 "Club" release can ship at its originally-designed scope.

## Lessons Learned & Gotchas

- **A high prompt-cache read ratio (97.9%) does NOT mean per-scan costs are low.** It only means the static text portion of each prompt is being cache-hit. Image tokens (Vision) and cache-WRITE costs on cold scans are still full price. The $0.015/scan figure in memory was misleading because it measured only the cached-text component. Future cost-baseline reads need to triangulate against actual $/scan from the Anthropic Cost dashboard, not derived from cache-hit ratios.
- **`CACHE_VERSION` bumps are an antipattern at this codebase's fix cadence.** When backend per-class fixes ship 2-5× per week (as they did in May), each global bump wipes the cache faster than it can rebuild. The new rule documented in `backend_cache_invalidation_pattern.md` is now: per-class fix = one line in `CLASS_INVALIDATIONS`, never a global bump.
- **The `/api/health` cache stats are in-memory and reset on every Render restart.** Useful for "did the process restart" check but the percentage is only meaningful after several hours of accumulated scan volume on a stable process. Don't read a fresh 20% hit rate as alarming — it's just a young counter.
