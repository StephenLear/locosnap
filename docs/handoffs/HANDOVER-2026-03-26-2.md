## What Worked
- Temperature 0 on vision calls is the correct fix for class oscillation — deterministic output from same input
- Responding to German comments in German built credibility and converted a critic (Nero) into a tester within the same thread
- Hardcoding known-correct ICE specs directly in the prompt is more reliable than relying on AI knowledge alone
- Flipping the Wikidata/AI mismatch rule immediately fixed the 300 km/h issue — structured data beats LLM on factual specs

## What Didn't Work (and Fixes)
- Frankfurt v2 showed BR 407 and 265 km/h — both wrong → replaced with v3 showing correct class and 320 km/h
- maxSpeed mismatch rule was discarding correct Wikidata values in favour of hallucinated AI values → flipped to trust Wikidata
- selectedIndex defaulting to 1 caused silent purchase failure when only 1 RevenueCat package returned → now finds annual index dynamically

## Key Decisions
- **Trust Wikidata over AI on speed mismatch**: Wikidata is a structured factual database — when it disagrees with AI by >20%, the AI is almost certainly wrong, not Wikidata
- **Hold frontend commit until build timing confirmed**: paywall fixes are ready but not committed — avoids partial builds
- **Respond to German comments in German**: showed respect for the audience and directly contributed to Nero converting
- **Play Store internal test link over direct APK**: cleaner experience, auto-updates, Play Store handles device compatibility

## Lessons Learned & Gotchas
- ICE family is a minefield — 403, 406, 407, 408, 462 look similar on camera. Temperature 0 alone won't fix misidentification between visually similar variants. Wikidata + hardcoded specs in prompt is the right layered approach.
- German rail enthusiasts are highly knowledgeable and will call out errors within the hour. This is actually an asset — they engage, they correct, and if handled well they become advocates.
- The mismatch rule was originally written to protect against stale Wikidata — but the real-world failure mode was the opposite. Document this for future prompt work.
- Replies to stephen@locosnap.app only reach Stephen if Hostinger forwarding is active — verify this is working for Nero's reply.
