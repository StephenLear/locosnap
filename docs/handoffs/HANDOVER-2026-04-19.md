## Lessons Learned & Gotchas

- **Before asking for any LocoSnap credential, read `backend/.env` and `memory/credentials_locations.md`.** Persisted this session — never re-ask Stephen to paste a key he's already provided
- **Before asking for a screen recording of an app card, grep backend specs/rarity files for the target class.** If no hardcoded rule exists, warn about likely AI hallucination before the recording happens. Pattern repeats: BR 143, BR 485, Dr18, Sr-class, BR 140 all had this issue
- **The 186-units hallucination was a clean leak from DB Class 156's real count.** When the AI doesn't know, it borrows specs from nearby classes. Any DE class that sounds even slightly like "rare / withdrawn / low-volume" risks getting Class 156's numbers. Guard explicitly in `trainSpecs.ts` for classes with confusable production counts
- **YouTube rail-footage uploaders routinely use loose hashtagging.** The "#BR140 #E50" combo on the DB Museum clip is typical — the uploader knows they're related, but the loco on screen is only ONE of them. Always check fleet number before trusting
- **Letterbox is not visually bad for app card reveals.** The black bars actually help focus the eye on the card and give room for bold subtitles. Don't default to full-frame when the card has content at top AND bottom
- **Render's real backend URL for LocoSnap is `https://locosnap.onrender.com`** — not `locosnap-api` or `locosnap-backend`. Confirm from `frontend/constants/api.ts` or `frontend/.env`
- **"Push-or-hold decision required after backend commit"** rule fired cleanly — user said "push", commit deployed, rescan confirmed. Good cadence
- **UK-only content pivot is confirmed underperforming.** Three levels of evidence now (TikTok views, TikTok retention, App Store daily downloads). Shift rotation heavily DE for the foreseeable future. UK retains value for brand diversity but not for monetisation
